# Cboe DataShop — Example Data Source

Source: [https://datashop.cboe.com/data-products](https://datashop.cboe.com/data-products)

Cboe DataShop offers sample data files covering equities, options, volatility, FX, futures, indices, and regulatory reporting. The onboarding agent crawls the DataShop page, downloads all available sample files (CSV, Excel, ZIP archives with extracted members), reads accompanying documentation (PDFs, data dictionaries), and creates **58 BigQuery tables** across two bronze datasets.

After onboarding, use **`agent_chat`** to ask questions across all three personas.

---

## Running the Questions

You can ask these questions interactively via `uv run adk web` (select `agent_chat`), or run all of them automatically and collect results:

```bash
# From the data-onboarding project root:

# Run all 30 questions (sequential, 5s delay between to avoid 429s)
uv run python examples/cboe/run_cboe_questions.py

# Run one persona at a time
uv run python examples/cboe/run_cboe_questions.py --persona "Data Analyst"
uv run python examples/cboe/run_cboe_questions.py --persona "Data Engineer"
uv run python examples/cboe/run_cboe_questions.py --persona "Catalog Explorer"

# Run a single question
uv run python examples/cboe/run_cboe_questions.py --id data-analyst-q3

# Resume (skip already-completed questions)
uv run python examples/cboe/run_cboe_questions.py --resume

# Adjust delay between questions (default: 5s)
uv run python examples/cboe/run_cboe_questions.py --delay 10
```

Results are saved incrementally to `examples/cboe/results/cboe_results.json`. After running, build the Results section of this document:

```bash
# Preview results markdown to stdout
uv run python examples/cboe/build_cboe_results.py

# Write results into this file (replaces everything after "## Results")
uv run python examples/cboe/build_cboe_results.py --write
```

**Answer:**

The implied volatility smile for the nearest expiration (2026-03-31) from the delta-relative volatility surface data shows a distinct downward slope, often called a "smirk."

**Key Insights:**
*   **Volatility Smirk:** Implied volatility is significantly higher for lower strike prices (representing out-of-the-money puts) compared to higher strike prices. For example, at a strike of 275.76, the IV is approximately 41.8%.
*   **Downside Protection Cost:** The higher IV at lower strikes suggests that the market is pricing in a higher probability of significant downward moves, reflecting the cost of downside protection.
*   **ATM Volatility:** Near the at-the-money (ATM) strike (around 509.42), the implied volatility is approximately 23.1%.
*   **Upside Skew:** For higher strikes (out-of-the-money calls), the implied volatility continues to decrease (e.g., ~15.3% at strike 679.30), indicating that the market assigns a lower relative probability to extreme upside moves compared to extreme downside ones.

The chart is available in the artifact.

**Answer:**

The schema decisions for the Volatility Surface tables involve four distinct table structures designed to handle intraday snapshots of implied volatility for underlying securities (primarily SPY). These tables are categorized by how they handle time (Constant Maturity vs. Expiration Specific) and strike prices (Delta Relative vs. Price Relative).

Here are the proposed schema decisions for each of the volatility surface tables:

### 1. `volatility_surfaces_constant_maturity_delta_relative`

*   **Reasoning:** Intraday snapshots of the implied volatility surface for a specific underlying (SPY), structured by constant maturity and constant delta. Each row represents a point in time for a specific maturity (e.g., 1M), showing the implied volatility and strike for various delta levels (e.g., 5-delta, 10-delta).
*   **Clustering:** Clustered by `underlying_symbol`.
*   **Key Columns and Notes:**
    *   `underlying_symbol` (STRING)
    *   `maturity` (STRING)
    *   `snapshot_time` (TIME): Only a time component. The date is in the filename. Consider combining them into a TIMESTAMP.
    *   `underlying_price` (FLOAT64)
    *   `iv_XX_delta` (FLOAT64): Implied volatility at various delta levels (e.g., `iv_05_delta`, `iv_50_delta`). The wide format is difficult to query; consider unpivoting into a long format.
    *   `strike_XX_delta` (FLOAT64): Strike price corresponding to various delta levels.
    *   `confidence_XX_delta` (FLOAT64): Confidence score for the calculation at various delta levels.

### 2. `volatility_surfaces_constant_maturity_price_relative`

*   **Reasoning:** Intraday snapshots of the implied volatility surface for a specific underlying (SPY), structured by constant maturity and strike price relative to the underlying price. Each row represents a point in time for a specific maturity (e.g., 1M), showing the implied volatility for various strike levels expressed as a percentage of the underlying price (e.g., 90% of spot, 110% of spot).
*   **Clustering:** Clustered by `underlying_symbol`.
*   **Key Columns and Notes:**
    *   `underlying_symbol` (STRING)
    *   `maturity` (STRING)
    *   `snapshot_time` (TIME): Only a time component. The date is in the filename. Consider combining them into a TIMESTAMP.
    *   `underlying_price` (FLOAT64)
    *   `iv_XX_price_relative` (FLOAT64): Implied volatility at various price relative levels (e.g., `iv_50_price`, `iv_100_price`). The wide format is difficult to query; consider unpivoting into a long format.
    *   `strike_XX_price_relative` (FLOAT64): Absolute strike price at various price relative levels.
    *   `confidence_XX_price_relative` (FLOAT64): Confidence score at various price relative levels. Contains negative values which likely represent error codes or missing data and should be handled.

### 3. `volatility_surfaces_expiration_specific_delta_relative`

*   **Reasoning:** Intraday snapshots of the implied volatility surface for a specific underlying (SPY), structured by specific expiration date and constant delta. This provides a raw, un-interpolated view of the volatility smile for each listed expiry.
*   **Clustering:** Clustered by `underlying_symbol` and `snapshot_time`.
*   **Partitioning:** Partitioned monthly by `expiration_date`.
*   **Key Columns and Notes:**
    *   `underlying_symbol` (STRING)
    *   `expiration_date` (DATE): Needs cleaning from `MM/DD/YYYY` string format.
    *   `snapshot_time` (TIME): Only a time component. The date is in the filename. Consider combining them into a TIMESTAMP.
    *   `underlying_price` (FLOAT64)
    *   `iv_XX_delta` (FLOAT64): Implied volatility at various delta levels. The wide format is difficult to query; consider unpivoting into a long format.
    *   `strike_XX_delta` (FLOAT64): Strike price corresponding to various delta levels.
    *   `confidence_XX_delta` (FLOAT64): Confidence score for the calculation at various delta levels.

### 4. `volatility_surfaces_expiration_specific_price_relative`

*   **Reasoning:** Intraday snapshots of the implied volatility surface for a specific underlying (SPY), structured by specific expiration date and strike price relative to the underlying price. This provides a raw, un-interpolated view of the volatility smile for each listed expiry against relative strike levels.
*   **Clustering:** Clustered by `underlying_symbol`.
*   **Partitioning:** Partitioned monthly by `expiration_date`.
*   **Key Columns and Notes:**
    *   `underlying_symbol` (STRING)
    *   `expiration_date` (DATE): Needs cleaning from `MM/DD/YYYY` string format.
    *   `snapshot_time` (TIME): Only a time component. The date is in the filename. Consider combining them into a TIMESTAMP.
    *   `underlying_price` (FLOAT64)
    *   `iv_XX_price_relative` (FLOAT64): Implied volatility at various price relative levels. The wide format is difficult to query; consider unpivoting into a long format.
    *   `strike_XX_price_relative` (FLOAT64): Absolute strike price at various price relative levels.
    *   `confidence_XX_price_relative` (FLOAT64): Confidence score at various price relative levels. Contains negative values which likely represent error codes or missing data and should be handled.

### Key Schema Design Decisions and Insights:

*   **Clustering Strategy:** All volatility surface tables are clustered by `underlying_symbol` (and sometimes `snapshot_time`) to optimize queries that filter by specific tickers.
*   **Handling "Wide" Format Data:** The source data is provided in a "wide" format (e.g., individual columns like `iv_05_delta`, `iv_10_delta`, etc.). The design notes highlight that this format is difficult to query and suggest that these tables should eventually be **unpivoted into a "long" format** (columns for `delta_level` and `implied_volatility`) to improve analytical flexibility.
*   **Timestamp Construction:** Because the source files often store the time of the snapshot separately from the date (which is encoded in the filename), the schema decisions recommend **combining the file date and snapshot time** into a single `TIMESTAMP` column for better temporal analysis.
*   **Data Cleaning:** Expiration dates provided in the source files as `MM/DD/YYYY` strings are flagged for transformation into standard BigQuery `DATE` types.
*   **Error Indicators in Confidence Scores:** Analysts noted that negative values (e.g., `-5.0`, `-6.0`) in the confidence columns serve as **error codes or indicators of missing data**, requiring special handling during data processing.
*   **Partitioning:** Tables for specific expirations (like `volatility_surfaces_expiration_specific_delta_relative`) are proposed to be **partitioned monthly by expiration date** to improve performance when analyzing specific options cycles.

**Answer:**

The four volatility surface tables in `data_onboarding_datashop_cboe_com_bronze` all provide intraday snapshots of the implied volatility surface for the underlying asset SPY. They are differentiated by two primary factors:

1.  **Maturity/Expiration Structure**:
    *   `volatility_surfaces_constant_maturity_delta_relative` and `volatility_surfaces_constant_maturity_price_relative` are structured by **constant maturity**. This means the data is presented for standardized timeframes to expiration (e.g., 1 month, 3 months), rather than specific calendar dates.
    *   `volatility_surfaces_expiration_specific_delta_relative` and `volatility_surfaces_expiration_specific_price_relative` are structured by **specific expiration date**. These tables show the raw, un-interpolated volatility smile for each actual listed expiry.

2.  **Volatility Surface Axis**:
    *   `volatility_surfaces_constant_maturity_delta_relative` and `volatility_surfaces_expiration_specific_delta_relative` present the volatility surface based on **constant delta**. This means they show implied volatility for various delta levels (a measure of option price sensitivity to changes in the underlying asset's price).
    *   `volatility_surfaces_constant_maturity_price_relative` and `volatility_surfaces_expiration_specific_price_relative` present the volatility surface based on **strike price relative to the underlying price**. This means they show implied volatility for various strike levels expressed as a percentage of the underlying price.

In essence, these four tables offer different perspectives on the same underlying concept of implied volatility surfaces, allowing users to analyze volatility across constant maturities or specific expiration dates, and across different option deltas or relative strike prices.
